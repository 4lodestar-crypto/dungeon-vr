# Ticket V0-001: Desktop-First Grid Movement + Server-Authoritative Tick Loop

**Version:** V0
**Priority:** Critical (blocks all other work)
**Owner:** Orchestrator coordinates; specialists assigned below
**Estimated effort:** 1–2 weeks part-time

---

## Goal

Establish the foundational architecture for Dungeon VR on Windows Desktop: a grid-based movement system driven by a server-authoritative tick loop, with procedural dungeon generation, so that Andrew can walk a champion through a generated dungeon using keyboard controls.

By the end of this ticket, Andrew should be able to:

1. Launch the Unity 6 LTS project on Windows Desktop
2. See a procedurally generated dungeon (rooms connected by corridors)
3. Use WASD to move a champion tile-by-tile across the dungeon
4. Turn 90° with A/D for rotations
5. See the champion blocked by walls (cannot walk through dungeon walls)
6. Observe the 20 Hz server-authoritative tick loop processing movement requests

## Why this ticket supersedes the original V0-001

The original `V0-001-hello-vr-world.md` targeted Meta Quest 3 VR: hand tracking, grabbable torch, pushable door, APK build. Per CLAUDE.md, V0–V1 targets **Windows Desktop only**. VR is deferred to V2+. This ticket re-scopes to the desktop-first strategy.

The original ticket is archived at `docs/tickets/archived/V0-001-hello-vr-world.superseded.md` for design history.

## Scope

**In scope:**
- Unity 6 LTS project setup for Windows Desktop standalone
- URP configured (default quality for desktop)
- Champion tile-based movement: 3m grid pitch, 4 cardinal directions, WASD input
- Server-authoritative tick loop: 20 Hz FixedUpdate, request-validate-apply-emit
- Procedural dungeon generation: random rooms + corridors with connectivity guarantee
- Grid collision: champion blocked by walls (bounds + wall check)
- First-person camera following champion facing direction
- EditMode unit tests for movement, tick loop, and grid collision
- GitHub Actions CI for Windows build + EditMode tests
- Windows Standalone build verification

**Out of scope (deferred):**
- Meta XR SDK, hand tracking, VR interaction — V2+
- Torch/door/"Hello Dungeon" scene elements (old V0-001) — V1+
- Level data format (JSON/ScriptableObject) — V1
- Monsters, AI, combat — V1+
- Free-locomotion movement — never (per CLAUDE.md)
- PlayMode tests — V1+
- APK/Android builds — V2+
- Art assets, materials, lighting — V1+

## Architectural exceptions (V0 only)

Per CLAUDE.md rule #1, every state change routes through the server layer. For V0, the tick system and movement handler run in-process with the `GameTick` MonoBehaviour as a thin wrapper. All V0-exception files bear the comment: `// V0-EXCEPTION: refactor through proper server layer in V1`.

This is the only allowed exception. From V1 onward the rule is absolute.
